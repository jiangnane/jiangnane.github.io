车机 AI Agent 平台产品

三月 24, 2026 HEIN

可以，而且这比“公司三年规划”更适合联通智网当前阶段：先聚焦做一个车机 AI Agent 产品，再用这个产品反推中间件、云服务和商业化能力。整体判断是：

方向是对的，符合座舱未来演进；车厂会感兴趣，但只会为“可量产、可控、可运营”的产品买单，不会只为一个聊天机器人买单。 智能座舱正从“功能座舱/被动交互”向“认知智能座舱/主动交互”演进，白皮书把 2022–2026、2027–2031 视为从网联化向更高阶智能化发展的阶段；同时，AI Agent 已被汽车基础软件白皮书明确视为 AIOS 时代的重要趋势，能够打通智驾、座舱、车控等多域能力。

我建议你们把产品定义为：

“车机 AI Agent 平台” = AI 入口 + Tool 调用层 + 车机中间件 + 云端服务运营。

这里的关键不是“像豆包手机那样会聊天”，而是像豆包手机那样能完成任务。腾讯的车载 AI 材料已经给了一个很接近的方向：AI Agent + 开放生态，自动闭环执行复杂操作，并与云端大模型和车企能力共建。

一、这个方向是否符合座舱未来发展方向

符合，而且是偏主线方向。 你上传的《汽车智能座舱分级与综合评价白皮书》把智能座舱技术体系拆成“三横三纵”，其中云平台、网联服务、人工智能、多模态交互、认知与决策技术都属于核心组成部分；它还明确描述了从 L0/L1 到 L3/L4 的演进，会从“授权/被动交互”走向“部分主动交互、全场景主动交互”。

《AI 加速汽车智能化跃迁》也直接把“AI Agent + 开放生态 + 主动服务 + 自动执行”作为座舱产品形态，强调“秒回”“端云结合”“本地 RAG”“常用常新”。这说明行业不只是把 AI 当成一个问答模型，而是当成新的座舱入口。

从更长周期看，华为鸿蒙座舱研究里也把座舱演进描述为：机械化 → 电子化 → 网联化 → 智能驾驶/第三生活空间，强调场景化主动服务与信息、娱乐、订餐、互联能力的融合。这个演进逻辑和“车机 AI Agent”高度一致。

二、市场和车厂接受度如何

市场端接受度：有需求，但用户更愿意为“省事、高频、刚需”买单，而不是为泛聊天买单。 你上传的《2025 中国多模态 AI 大模型座舱应用洞察》显示，用户对 AI 行程规划、智能推荐、停车助手、用车服务等功能更关注，而对一些更炫但低频的能力会更保守。不同人群对识别类、监测类、服务类功能接受度也不一样，但“用车服务类”和“效率提升类”更接近真实付费场景。

车厂端接受度：会接受，但前提是你卖的是“可量产产品”，不是“Agent Demo”。 《汽车人工智能应用案例集》已经列出多个座舱智能化案例，比如东风的多模态座舱认知智能、岚图基于大模型 Agent 的语音交互、五菱灵语 AI、大模型智能座舱 OS 等，说明主机厂和供应商已经在上车验证。 《基础软件发展研究报告 6.0》还明确建议推动 Tier1 向“软件+硬件服务商”转型，建立软件认证、模拟测试环境、安全模板和统一接口体系，说明 OEM 不是不接受，而是希望 Tier1 能提供工程化、标准化、可验证的方案。

换句话说，车厂接受的是：

  1. 能提升差异化体验
  2. 能复用到多车型
  3. 能接进现有座舱架构
  4. 安全、权限、数据合规可控
  5. 能带来后续运营收入

三、联通智网做这个产品，最该研发哪些能力

我建议按四层来做，而不是一上来先做“大模型”。

1）最先做的是：车机中间件能力层

这是你们真正的护城河。 你前面已经定义得很清楚：中间件位于 Android App 和 Framework 之间，目的是抹平 Framework API、供应商能力、Android Automotive 车载能力的差异，同时暴露跨域访问接口和 AI 可调用接口。这个方向非常合理，也和基础软件白皮书里“中间件层承担跨层适配角色、应用层直接调用场景接口”的思路一致，但你们要明确与 SOME/IP/DDS、Autosar、虚拟化那类 OS/域间中间件区分开。

这一层建议研发 5 类能力：

  • 统一 Vehicle Capability API：车控、空调、座椅、车窗、氛围灯、仪表、语音状态等
  • 统一 App/Service Capability API：导航、媒体、电话、短信、账户、日历、消息
  • Vendor 适配层：封装供应商 jar、系统服务差异、权限差异
  • AI Tool API：把车机能力包装成 Tool/Function Calling 接口
  • 跨域访问层：车机 ↔ 云端 ↔ 车联网平台 ↔ 第三方服务

这层做出来，才谈得上“多车厂复用”和“避免再次被硬件供应商绑架”。

2）第二层做：Agent 编排与工具调度能力

这是产品的大脑，但不一定是你们全部自研。 需要研发的是：

  • 意图理解和任务拆解
  • Tool 选择与执行编排
  • 多步任务状态机
  • 错误恢复和兜底策略
  • 记忆与上下文管理
  • 用户画像与偏好系统
  • 结果确认与人机交互闭环

这部分可以大量借开源框架，不一定自己从零写。

3）第三层做：车生活与内容服务能力

这是未来营收来源。 要优先研发能形成真实闭环的服务，而不是泛功能堆砌。建议优先：

  • AI 行程规划
  • 充电/停车/加油/洗车
  • 音乐/视频/儿童内容
  • 会员、流量包、权益包
  • 车主服务提醒（保养、违章、保险、年检）
  • 本地生活服务（餐饮、商超、景区）

这和你上传的用户功能偏好研究是一致的：服务类与效率类更容易转化。

4）第四层做：工程化与量产能力

这个往往比模型更重要。需要补齐：

  • 模拟测试环境
  • Tool 安全沙箱
  • 车规权限模型
  • 端云协同策略
  • 离线兜底与弱网降级
  • 日志、埋点、评测、A/B 实验
  • 模型/工具 OTA
  • 内容合规与数据合规

白皮书里专门提到 AI 模型 OTA、跨域统一接口、模拟测试环境、安全认证模板，这些都说明车厂会把工程化能力当成采购门槛。

四、建议做成什么产品，不建议做成什么产品

建议做成： 一个“车机 AI Agent 平台产品”，包含 3 个最小闭环：

  1. AI 车控助手 说一句话就能完成车控、导航、媒体切换、场景联动。

  2. AI 行程与车生活助手 从“我周末想去露营”开始，自动做路线、充电、停车、餐饮、天气、提醒。

  3. AI 服务运营入口 承接流量、会员、内容、服务分成，和联通智网的运营能力打通。

不建议做成： 一个“像豆包一样的车载聊天 App”。 因为那样很容易被车厂认为“可替代”“价值低”“没法量产落地”。

五、为了实现营收目标，实施路径怎么走

如果只聚焦这个产品，我建议走 “产品先行、平台沉淀、运营变现” 的三段式，周期控制在 3 年内。

第 1 阶段：0–9 个月，做出可演示、可试点的 MVP

目标不是赚钱，而是证明“可落地”。

产品范围建议只做 3 个闭环：

  • AI 车控助手
  • AI 行程规划
  • AI 车生活服务入口（停车/充电/餐饮三选二）

这阶段的交付物应该是：

  • 车机中间件 MVP
  • Tool API 规范
  • Demo 车/原型系统
  • 1 套端云协同架构
  • 1 套可展示的运营后台

这阶段核心 KPI 不是收入，而是：

  • 1–2 家车厂 PoC
  • 1 套参考硬件平台
  • 20–30 个可调用 Tool
  • 首屏延迟、任务完成率、弱网兜底可被展示

第 2 阶段：9–18 个月，做成可量产产品包

目标是拿到项目，不一定先拿大规模订阅收入。

把产品打包成三件套：

  • 车机 AI Agent SDK / 平台包
  • 车生活服务包
  • 车厂运营后台

商业模式上，这阶段以：

  • 平台接入费
  • 项目开发费
  • 年度维护费
  • 联合创新费

为主。 不要急着靠订阅冲规模，因为车厂第一年更容易接受“项目预算”，不容易一下子接受高比例分成。

第 3 阶段：18–36 个月，做运营变现

目标是让产品开始带来持续收入。

优先变现的不是模型，而是服务：

  • 流量包/热点包
  • 内容会员
  • 停车、充电、洗车、保养分成
  • AI 高级助手包
  • 车厂用户运营 SaaS

这个路径非常适合联通智网，因为运营商资源、流量运营、车联网连接、用户运营恰好都是现成优势。

六、可以参考哪些开源项目

我建议分 6 类选型，不要只盯一个 AutoGLM。

1）Agent 框架

如果想快速做原型，首选这些：

  • Open-AutoGLM：开源的 Phone Agent 模型与框架，核心是“看屏幕 + 规划 + 自动操作”，非常适合借鉴“像豆包手机那样的 Agent 交互范式”，但原生面向手机，不是车机量产框架，需要你们自己替换成车机 Tool 调用模式。(GitHub)
  • Qwen-Agent：支持 Function Calling、RAG、MCP，比较适合国内模型栈和自部署。(GitHub)
  • LangChain / LangGraph：生态最成熟，适合快速搭建工具编排和复杂工作流。(GitHub)
  • LlamaIndex：更适合做知识增强、RAG、车主手册/FAQ/服务知识查询。(GitHub)
  • Semantic Kernel：企业级、多代理编排更稳，适合后期平台化。(GitHub)
  • AutoGen：多 Agent 协作强，但更偏研究和复杂编排，不一定适合第一版产品。(GitHub)

2）Tool 协议与开放生态

  • MCP（Model Context Protocol）：适合把工具调用标准化，未来你们如果要开放给第三方服务商，MCP 很值得作为工具适配层参考。(GitHub)

3）AAOS / 车机开发样例

  • android/car-samples：Android 官方车载样例，适合做导航、停车、充电等车载 App 模板。(GitHub)
  • android/car-codelabs：官方 codelab，适合团队快速上手 Android for Cars。(GitHub)
  • volvo-cars/automotive-media-sample：AAOS 媒体应用样例，适合媒体/内容能力开发。(GitHub)
  • COVESA aosp_device_covesa_emulator：适合第三方应用和平台原型验证。(GitHub)

4）语音与端侧基础能力

  • whisper.cpp：端侧 ASR 很实用，适合低算力或离线兜底。(GitHub)
  • 你们也可以考虑搭配 FunASR、Sherpa-ONNX 一类端侧语音项目，但这次我没有去网上逐个核最新状态，就不展开硬性推荐了。

5）车机/车载 UI 参考

  • BricksOpenLauncher：可参考车载 Launcher 交互。(GitHub)
  • pilot-drive：偏 DIY 车机，但适合理解模块化车机服务设计。(GitHub)

6）地图与导航

  • MapLibre Android Auto Sample：如果你们不想强绑定商业地图 SDK，这个样例可参考。(GitHub)

七、AutoGLM 适不适合作为你们的核心选型

适合做原型和 Demo，不适合直接当量产核心。

原因很简单：

  • 它的“Phone Agent”思路和你们要做的“豆包式车机 Agent”很接近，适合快速验证交互范式。(GitHub)
  • 但它原生是“手机屏幕理解 + ADB 自动操作”的范式，而车机量产更需要“显式 Tool 调用 + 权限控制 + 安全审计 + 车规沙箱”。

所以我的建议是：

  • MVP 阶段：可以用 Open-AutoGLM / Qwen-Agent 快速做交互原型
  • 产品化阶段：逐步切到“你们自有中间件 + Tool API + 可替换 Agent 框架”的架构

也就是：

不要把 AutoGLM 当产品本身，要把它当验证产品方向的脚手架。

八、车厂真正会关心的卖点是什么

如果你拿这个产品去和整车厂沟通，建议不要讲“我们有大模型”，而是讲 4 个卖点：

1）更高效的人机交互

从“菜单点击”变成“任务完成”，减少 HMI 层级和驾驶干扰。 这符合白皮书里从被动交互到主动交互、从功能座舱到认知智能座舱的演进。

2）更强的平台复用能力

通过中间件抹平不同 SoC、不同供应商能力差异，降低新车型适配成本。 这也是你们最应该主打的 To B 价值。

3）更好的运营变现能力

不是卖模型，而是卖：

  • 车生活服务
  • 流量/会员
  • 内容服务
  • 用户运营能力

这正是联通智网最有差异化的地方。

4）更符合 AI 座舱趋势

行业已经在推动 AI Agent、跨域数据流转、统一接口、模型 OTA、端云协同。你们不是在“探索一个边缘方向”，而是在做主航道上的产品。

九、最终建议：这件事该不该做

该做，而且建议现在就按产品来立项。

但立项方式不要叫：

  • “大模型上车”
  • “车机豆包”
  • “车载聊天机器人”

而是叫：

“车机 AI Agent 平台产品”

产品目标也不要写成“会对话”，而要写成：

  • 完成任务
  • 接入车控
  • 接入车生活服务
  • 接入运营体系
  • 可跨车型复用

一句话总结：

这个方向符合座舱未来发展，市场端有需求，车厂端有兴趣；真正的成败不在模型，而在中间件、Tool 体系、工程化能力和运营闭环。
