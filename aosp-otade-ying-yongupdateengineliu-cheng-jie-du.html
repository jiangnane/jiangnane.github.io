AOSP OTA的应用：UpdateEngine流程解读

九月 23, 2024 HEIN

1. UpdateEngine 简介

UpdateEngine 是 Android OTA (Over-The-Air) 更新的核心模块，它负责安全高效地下载、验证并应用系统更新。在 Android 设备升级过程中，UpdateEngine 保证了设备不会因为更新失败而损坏，这通过 A/B 分区机制完成。

2. OTA 更新方式

2.1 完整更新 (Full OTA)

完整更新会替换系统的所有文件和分区。这适用于系统大版本更新，尽管包比较大，但能够确保完整性和一致性。

2.2 增量更新 (Delta OTA)

增量更新只下载相对于旧版本变化的部分，大大减小了更新包的大小，节省了下载时间和流量。这种更新方式是 UpdateEngine 的重要特性。

3. A/B 分区机制

A/B 分区是 Android 引入的一项技术，它将系统镜像分为两个分区 (slot A 和 slot B)。在更新过程中，UpdateEngine 会将更新应用到非活动分区。例如，当 slot A 正在运行时，更新包会写入 slot B，更新完成后，设备重启并从 slot B 启动。

3.1 好处

  • 无缝更新：更新发生在后台，不影响用户使用设备。
  • 回滚机制：如果更新失败，设备可以回滚到之前的状态，确保不会变砖。

4. 工作流程

以下是 UpdateEngine 处理 OTA 更新的流程：

  1. 下载更新包：通过网络下载增量或完整更新包。
  2. 验证更新包：在解包之前，UpdateEngine 通过校验签名和哈希值来验证更新包的真实性和完整性。
  3. 解压并应用更新：使用 PayloadProcessor 解压并将更新包写入到备用分区中。
  4. 切换分区：在更新完成后，系统通过重启切换到已更新的分区。
  5. 验证更新：更新后，系统会进行自检，确保更新已成功应用，未检测到问题时，旧分区将被标记为可以重用。

5. 关键模块

UpdateEngine 的设计是模块化的，主要包括以下几个部分：

5.1 UpdateEngineClient

这是 UpdateEngine 的接口，用于和 Android 系统的其他组件（如系统更新界面）进行交互。它允许启动更新、监控进度以及提供回调信息。

class UpdateEngineClient {
  // Starts an update process with a given URL for the OTA package.
  void StartUpdate(const std::string& update_url);
  
  // Provides progress updates on the OTA process.
  void OnProgressUpdate(double progress);
  
  // Notifies the client when the update is completed or failed.
  void OnComplete(bool success);
};

5.2 PayloadProcessor

该模块处理 OTA 包的解包与更新内容的写入工作。它将下载的更新包拆解并应用到目标分区。

bool PayloadProcessor::ApplyPayload(const std::string& payload_path, ...);

5.3 分区验证 (dm-verity)

Android 使用 dm-verity 技术来保证系统分区在运行时没有被篡改。UpdateEngine 通过这一机制验证更新后的分区的完整性。

6. A/B分区的增量更新

增量更新包中的数据是基于现有的系统文件生成的。如果在 slot B 中没有 slot A 的系统文件，增量更新无法直接应用，因为它依赖于对旧文件的差异计算。

为了解决这个问题，A/B 更新机制会先确保当前活动分区（如 slot A）中的文件完好无损，然后将增量更新包应用到 slot B。这是因为增量更新包只包含相对 slot A 的变化，而不是整个系统。如果 slot A 缺失文件，更新将失败。

6.1 增量更新关键步骤：

  1. 验证 slot A 文件的完整性：在应用增量更新之前，系统首先检查 slot A 是否有完整的系统文件。如果文件丢失或损坏，系统将无法继续应用增量更新。

  2. 差异计算与文件修补：增量更新包的内容基于 slot A 当前文件与目标版本的差异，PayloadProcessor 通过计算差异来修补 slot B。

  3. 复制缺失文件：如果 slot B 缺少 slot A 中的重要文件，系统会从 slot A 复制必要的文件，确保 slot B 能够接收增量更新。

6.3 核心代码流程

在实际代码中，PayloadProcessor 负责对系统分区进行修补。具体操作步骤如下：

// Step 1: 验证 slot A 的文件是否完整
if (!VerifySlotAIntegrity()) {
    LOG(ERROR) << "Slot A missing critical files, cannot apply delta update.";
    return false;
}

// Step 2: 从 slot A 复制文件到 slot B (如果需要)
if (IsFileMissingInSlotB()) {
    CopyFileFromSlotAtoB("system_file");
}

// Step 3: 应用增量更新到 slot B
ApplyDeltaUpdateToSlotB("delta_update.bin");

7. 安全性与回滚

UpdateEngine 提供了丰富的安全特性，确保 OTA 更新过程中的数据安全：

  • 签名验证：每个更新包必须经过签名，防止恶意包被执行。
  • 完整性检查：更新包被验证哈希值，确保传输过程中数据没有损坏。
  • 回滚机制：如果系统在新分区启动失败，会自动回滚到旧分区，保证设备正常运行。

8. 日志记录与调试

UpdateEngine 会详细记录更新过程中的日志，帮助开发者调试更新失败或其他问题。通过以下命令可以查看日志：

adb logcat | grep update_engine

常见问题如更新包损坏、签名无效等都可以从日志中找到相关信息。

9. 应用实例

UpdateEngine 是大多数 Android 设备升级的核心部分。无论是每月的安全补丁，还是年度的 Android 大版本更新，用户通常通过 OTA 更新接收到这些变更。例如，Pixel 设备采用 A/B 分区机制，提供无缝更新体验，而其他厂商（如三星、华为）则有自己定制的更新实现。
